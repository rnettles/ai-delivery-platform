# ADR-026: Platform-Owned AI Governance Content

## Status
Accepted

## Date
2026-04-19

## Deciders
- Product Engineering
- Platform Architecture

---

## Context

The platform requires AI governance content to guide role execution and human agent behavior across managed projects. Prior to this decision, governance content for human agents was sourced by developers manually cloning or copying from `ai-project_template` — an external template repository containing `ai_dev_stack/ai_guidance/` files.

This approach has several problems:

1. **No lineage or auditability** — content cloned from a template is immediately disconnected from its origin. There is no way to determine which version of the template was used, whether it has drifted, or whether it complies with current platform rules.

2. **Manual, error-prone process** — developers must remember to clone, adapt, and keep the template-derived content up to date. This relies on convention rather than enforcement.

3. **Violates ADR-001** (Git as source of truth) — the platform has no authoritative record of which governance artifacts are active in a given project.

4. **Violates ADR-004** (Governance-first architecture) — governance content is produced outside the governed execution model and cannot be traced to a pipeline output.

5. **Inconsistency between human and automated agents** — the Execution Service loads governance at runtime from `platform/governance/` (ADR-025 Tier 2), while human agents in VSCode load governance from manually maintained per-project files. These two sources can drift without detection.

The Two-Tier Governance Composition Model (ADR-025) established that:
- Universal rules live in dotfiles (Tier 1)
- Platform-specific governance for automated roles lives in `platform/governance/` (Tier 2)
- Per-project governance artifact generation is a pipeline output responsibility of the sprint-controller role

This ADR makes the supersession of `ai-project_template` as the governance source explicit and final.

---

## Decision

The platform SHALL be the authoritative owner of AI governance content for all projects it manages.

### Supersession

`ai-project_template` SHALL NOT be used as the source for per-project AI governance content in platform-managed projects. It is superseded by:

1. **Tier 1 (dotfiles)** — Universal governance applying to all AI agents in all workspaces
2. **Tier 2 (`platform/governance/`)** — Platform-specific governance for automated pipeline roles
3. **Platform-generated per-project artifacts** — Generated by the sprint-controller role from `platform/governance/templates/`, deposited into the project repository as governed pipeline output

### Per-Project Governance Generation

The sprint-controller role SHALL generate the project's `ai_dev_stack/ai_guidance/` directory contents during project onboarding and governance refresh cycles, using templates from `platform/governance/templates/`.

This creates a traceable lineage:

```
platform/governance/templates/
  └─(sprint-controller execution)→ project/ai_dev_stack/ai_guidance/
      AI_CONTEXT.md
      AI_RULES.md
      AI_RUNTIME_POLICY.md
```

Generated artifacts SHALL be:
- Committed to the project repository as a governed artifact (ADR-001, ADR-002)
- Traceable to the sprint-controller execution record that produced them (ADR-019)
- Versioned alongside the platform governance version in effect at generation time

### Governance Drift Detection

When `platform/governance/` is updated, the sprint-controller SHALL flag projects whose `ai_dev_stack/ai_guidance/` was generated from an older template version, triggering a governance refresh cycle.

---

## Consequences

### Positive

- Per-project AI governance is auditable, traceable, and reproducible
- Human agents operating in a project workspace receive governance content produced and versioned by the platform, ensuring consistency with automated roles
- Eliminates the manual `ai-project_template` clone process
- Governance version drift is detectable across all managed projects

### Negative

- Projects not yet onboarded to the platform will continue using manually maintained governance content until the sprint-controller onboarding execution runs
- Template-generated artifacts add a pipeline step to project onboarding

### Neutral

- `ai-project_template` may be retained as a reference or bootstrap tool for projects not managed by the platform, but it has no authority over platform-managed projects
