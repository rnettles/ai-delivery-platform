# Design Document: API Layer for Execution & State Management

---

## 1. Overview

This document defines the **API Layer** that sits at the center of the system, enabling:

* Execution of deterministic scripts
* Shared state management across:

  * n8n workflows
  * Local development environments
  * Future agent systems
* Decoupling from underlying storage (Postgres, Blob, etc.)

---

## 2. Goals

### Primary Goals

* Provide a **single API interface** for:

  * execution
  * state management
* Eliminate **direct database access** from external systems
* Enable **shared memory across human + AI workflows**
* Support **scalability, replayability, and observability**

---

### Non-Goals

* Not responsible for:

  * business logic of scripts (delegated to execution service)
  * canonical artifact storage (handled by Git)
  * UI concerns

---

## 3. High-Level Architecture

```text
Clients (n8n, VSCode, Agents)
            ↓
        API Layer
     ┌───────────────┐
     │ Execution API │
     │ State API     │
     └───────────────┘
            ↓
     Internal Services
     ├─ Execution Service
     ├─ State Service
            ↓
     Storage Layer
     ├─ Postgres (state)
     ├─ Blob Storage (artifacts)
     └─ Git (canonical)
```

---

## 4. Core Concepts

---

### 4.1 Execution

Represents a **deterministic unit of work**

* Identified by `execution_id`
* Executes a named `script`
* Produces output and/or artifacts

---

### 4.2 State

Represents **shared memory** across workflows

| Type      | Description         |
| --------- | ------------------- |
| execution | runtime tracking    |
| workflow  | agent flow state    |
| session   | user/system session |

---

### 4.3 Artifact

Output generated by execution:

* JSON
* Markdown
* Files

Stored externally (Blob Storage)

---

## 5. API Surface

---

## 5.1 Execution API

---

### POST `/execute`

#### Request

```json
{
  "execution_id": "optional",
  "script": "string",
  "input": {},
  "metadata": {}
}
```

#### Response

```json
{
  "execution_id": "string",
  "status": "completed | failed",
  "output": {},
  "artifacts": [],
  "started_at": "ISO",
  "completed_at": "ISO"
}
```

---

### GET `/execution/:id`

Returns execution details

---

### GET `/scripts`

List available scripts

---

## 5.2 State API

---

### POST `/state`

Create or overwrite state

```json
{
  "state_id": "optional",
  "type": "workflow | agent | session",
  "scope": "planner | sprint | execution",
  "data": {},
  "metadata": {}
}
```

---

### GET `/state/:id`

Retrieve state

---

### PATCH `/state/:id`

Partial update

```json
{
  "data": {}
}
```

---

### POST `/state/query`

Query state records

```json
{
  "type": "workflow",
  "filters": {}
}
```

---

### DELETE `/state/:id`

Archive or delete

---

## 6. Data Model

---

### 6.1 Execution Model

```json
{
  "execution_id": "uuid",
  "script": "string",
  "status": "pending | completed | failed",
  "input": {},
  "output": {},
  "started_at": "timestamp",
  "completed_at": "timestamp",
  "metadata": {}
}
```

---

### 6.2 State Model

```json
{
  "state_id": "uuid",
  "type": "workflow",
  "scope": "planner",
  "data": {},
  "metadata": {},
  "version": 1,
  "created_at": "timestamp",
  "updated_at": "timestamp"
}
```

---

## 7. Storage Strategy

---

### Postgres

* JSONB columns for flexibility
* Indexed fields:

  * type
  * scope
  * metadata.source
  * created_at

---

### Blob Storage

* Stores large artifacts
* API returns references (URLs)

---

### Git

* Canonical source of truth
* Stores:

  * plans
  * schemas
  * final artifacts

---

## 8. Execution Flow

---

### Example Flow

```text
1. Client → POST /execute
2. Execution Service runs script
3. Result stored in State API
4. Artifacts stored in Blob
5. Response returned
```

---

## 9. State Flow

---

### Example Flow

```text
1. n8n → POST /state
2. Update workflow state
3. Later → GET /state/:id
4. Continue workflow execution
```

---

## 10. Versioning Strategy

* Each state record includes:

  * `version`
* Updates increment version
* Optional future:

  * full snapshot history
  * diff tracking

---

## 11. Security Considerations

* API authentication (future):

  * token-based
* Role-based access (future)
* No direct DB access externally

---

## 12. Observability

* Structured logs:

  * execution_id
  * state_id
* Metrics:

  * execution time
  * error rates

---

## 13. Scalability

* Stateless API layer
* Horizontal scaling via Container Apps
* DB scaling handled independently

---

## 14. Future Enhancements

* Event-driven architecture (event bus)
* Redis caching layer
* CKS integration for knowledge graph
* Workflow replay engine

## 15. Trade-offs

---

### Pros

* Decoupled architecture
* Agent interoperability

---

### Cons

* Added complexity
* Requires schema governance
* Potential central bottleneck
